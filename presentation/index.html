Module federation

Multiple separate builds should form a single application. These separate builds act like containers and can expose and consume code between builds, creating a single, unified application.
Isomorphic

Concepts

  • Webpack will provide a file commonly named remoteEntry.js that have the code to init container, init the sharing scope (dependancy versions resolutions) and get the exposed modules.
  • Share deps, services, state library as singleton(not recommended, conflicts)
  • Host will consume remoteEntry.js & exposed modules
  • Remote will expose remoteEntry.js & exposed modules in chunks
  • Application can be Host & Remote & standalone in same time. Circle dependancy work as well.

Use cases

  • Microfrontend
  • Components libs
  • A/B testing
  • Migration legacy, technical dept on framwork version, Mix technologies
  • Decoupling, Boundaries, Build performance.
  • Microfront end architecture
  • Components libs without the need to pull updates & rebuild in every projects
  • A/B testing, choose dynamically from a back end / a param url which container you import for a feature or a component
  • Migration: Incremental updates for big technical debt issues (framework version, legacy app...)
  • Decoupling, Build performance. Work in isolation & build small chunk of an application

Organization

Can reflect intern organization. Team own a domain with features. Can deploy independantly, application is up to date without a rebuild step
